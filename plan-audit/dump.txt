WHAT IS MARVIS?
---------------
Marvis is an AI orchestrator that:
  1. Accepts a natural-language task from you ("Write a tweet…")
  2. Finds the right specialist agent — from its MARKETPLACE (rented, pay-per-use)
     or from its own OWNED library (built once, runs free forever after)
  3. If NO skill covers the request (a genuine capability gap), Marvis hires a
     SkillBuilder to COMMISSION a brand-new specialist — a one-time purchase —
     then owns that specialist outright from then on
  4. Shows you the price, waits for your PIN to authorise payment
  5. Puts funds in ESCROW (never paid upfront)
  6. Dispatches the specialist (local Gemma via Ollama)
  7. Verifies the work deterministically (char limits, doc created, etc.)
  8. Shows you the result + advisory AI score
  9. Waits for a second PIN to release the completion payment (market hires only —
     owned-skill runs are free and skip this entirely)
 10. Settles the escrow, emits a receipt

Phase 1 = hire → pay → dispatch → verify → settle, for a fixed marketplace of skills.
Phase 2 = when the marketplace has a genuine hole, Marvis SELF-EXTENDS: it commissions
          a new skill once, then serves that platform for free forever after.

OWNED LIBRARY (skills Marvis BUILT and now runs for FREE)
-------------------------------------------------------------
  Starts EMPTY on a clean checkout. Populated only by a successful build.
  Lives on disk at:  app/owned-skills/<slug>/{skill.json, instruction.md}
  Survives a restart (re-seeded at startup) — but is invisible to the broker's
  /.well-known/skills forever; Marvis owns it, it is never for sale.

  Check what Marvis currently owns:
    ls app/owned-skills/

  To reset the demo and see the FULL gap→build flow from scratch:
    rm -rf app/owned-skills/linkedin-writer
    (then restart ./run_marvis.sh)

  DEMO 2 — SELF-EXTENDING BUILD (LinkedIn, Phase 2 flagship)
--------------------------------------------------------------------------------

Type (assuming app/owned-skills/ has been reset per above, or is fresh):
  ┌─────────────────────────────────────────────────────────────────────┐
  │  Write a LinkedIn post about my Marvis launch and save it as a      │
  │  Google Doc.                                                        │
  └─────────────────────────────────────────────────────────────────────┘

  Marvis parses the task, then discovers there is NO skill for LinkedIn in
  EITHER the marketplace or the owned library:
    Agent: "Capability gap: no skill in the marketplace or owned library
            covers linkedin (writer). Proposing a build."

  A BUILD-APPROVAL CARD appears — NOT a normal hire card:
  ┌───────────────────────────────────────────────────────┐
  │  No Skill Yet — Commission One?                       │
  │  No skill covers LinkedIn yet. Commission a            │
  │  LinkedinWriter you'll OWN? After this, all LinkedIn   │
  │  writer tasks run FREE forever.                        │
  │  ────────────────────────────────────────────────      │
  │  Build base fee:            $1.00   NON-REFUNDABLE    │
  │  On delivery (valid build):  $1.50   ONE-TIME PURCHASE │
  │  ────────────────────────────────────────────────      │
  │  Total held in escrow:      $2.50                     │
  │  Enter PIN to authorize the build                      │
  └───────────────────────────────────────────────────────┘

  PIN modal → enter your PIN → confirm. What happens next, all automatic:
    1. $2.50 moves user → escrow:build-<task_id>
    2. SkillBuilder (Gemma) writes a brand-new specialist's instruction text
       — Marvis itself derives the skill_id, agent_name, allowed tool
       (create_doc), specialties, and pricing; the model NEVER chooses those,
       so it can never fabricate a nonexistent tool.
    3. A deterministic validation gate checks the new skill for a fabricated
       tool, a duplicate id, or a generic instruction — reject → refund
       ($1.50 completion refunded, $1.00 base kept, nothing persisted).
    4. On PASS: $2.50 releases to agent:SkillBuilder, escrow closes.
    5. The new "LinkedinPostSpecialist" skill is written to
       app/owned-skills/linkedin-writer/ and registered — permanently.
    6. A live card appears: "✅ New Owned Skill — LinkedinPostSpecialist
       added to your owned skill library."
    7. Marvis immediately RE-RUNS your original request. It now finds the
       skill in the OWNED store — NO further payment, NO PIN prompt. The
       specialist writes the post and calls create_doc for free.
    8. Receipt card shows the delivered LinkedIn post. Total paid this turn:
       $2.50 (the one-time build) — the task itself was free.

  Now type the SAME kind of request again in a new message:
  ┌─────────────────────────────────────────────────────────────────────┐
  │  Write another LinkedIn post about our product roadmap.             │
  └─────────────────────────────────────────────────────────────────────┘

  This time: NO build card, NO PIN prompt at all. Marvis selects the owned
  LinkedinPostSpecialist directly, writes and saves the post, and shows the
  receipt — completely free. Wallet balance is unchanged.

Response:

--- STEP 4: Check balance ---
curl "http://localhost:8000/wallet/balance?token=<YOUR_TOKEN>"

--- STEP 5: Create ADK session ---
curl -X POST http://localhost:8000/adk-sessions \
  -H 'Content-Type: application/json' \
  -d '{"token":"<YOUR_TOKEN>"}'

Response:
{ "adk_session_id": "61bc9b5b64a740d488b225aca96e7e14", "user_id": "<YOUR_USER_ID>" }

--- STEP 6: Send your task (SSE stream) ---
curl -N -X POST http://localhost:8000/run_sse \
  -H 'Content-Type: application/json' \
  -d '{
    "app_name": "app",
    "user_id": "<YOUR_USER_ID>",
    "session_id": "<SESSION_ID>",
    "streaming": true,
    "new_message": {
      "role": "user",
      "parts": [{"text": "Write a LinkedIn post about my Marvis launch and save it as a Google Doc."}]
    }
  }'

The stream emits one SSE "data: {...}" line per event. Watch content.parts[]:
  - text containing <mstat>...</mstat>   → human-readable status line
  - text containing <a2ui-json>...</a2ui-json> → a UI card (hire/build/payout/receipt)
  - a functionCall named "adk_request_input" → a PIN gate has fired; its "id"
    field is the interrupt_id you must use to resume (see next step)

--- STEP 7: When a PIN gate fires, verify the PIN ---
The functionCall looks like:
  { "functionCall": { "name": "adk_request_input", "id": "build_auth", ... } }

interrupt_id is one of:
  "payment_auth"  — PIN #1, hiring a MARKETPLACE skill (Phase 1, unchanged)
  "build_auth"    — PIN #B, commissioning the SkillBuilder (Phase 2, new)
  "payout_auth"   — PIN #2, releasing a market hire's completion fee
                    (owned-skill runs have NO PIN #2 — nothing to release)

curl -X POST http://localhost:8000/auth/verify-pin \
  -H 'Content-Type: application/json' \
  -d '{
    "token": "<YOUR_TOKEN>",
    "pin": "1234",
    "adk_session_id": "<SESSION_ID>",
    "interrupt_id": "build_auth"
  }'

Response: { "ok": true }

--- STEP 8: Resume the stream ---
curl -N -X POST http://localhost:8000/run_sse \
  -H 'Content-Type: application/json' \
  -d '{
    "app_name": "app",
    "user_id": "<YOUR_USER_ID>",
    "session_id": "<SESSION_ID>",
    "streaming": true,
    "new_message": {
      "role": "user",
      "parts": [{
        "functionResponse": {
          "id": "build_auth",
          "name": "adk_request_input",
          "response": {"result": "confirmed"}
        }
      }]
    }
  }'

  (Use "response": {"result": "cancelled"} to reject instead — no PIN verify
  call is needed for a cancel.)

  Keep repeating STEP 7 → STEP 8 for every interrupt_id the stream emits,
  in order, until the stream ends on a terminal card (receipt / refunded /
  cancelled / output_failed).

   Same trigger, but on a FREE owned run.
   → Nothing to refund (nothing was ever paid) — retry the same request at no cost.

   If the builder ever produced a fabricated tool or a duplicate skill id, the
   deterministic validation gate rejects it before anything is persisted:
   build completion fee ($1.50) refunded, build base fee ($1.00) kept,
   nothing written to app/owned-skills/.

================================================================================
  WHAT EACH SPECIALIST CAN (AND CANNOT) DO
================================================================================

  Every specialist — marketplace OR owned — gets EXACTLY ONE MCP tool, chosen
  from a closed set of two: create_doc (write) / get_doc_content (read).
  A CapabilityGrant is minted fresh per task, expires in 5 minutes, and is
  revoked immediately after the specialist finishes — whether the run
  succeeded, failed, or (for the builder) never needed a tool at all.

  TwitterPostSpecialist / InstaPostSpecialist / LinkedinPostSpecialist (owned):
    ✅ ALLOWED:   create_doc
    ❌ BLOCKED:   get_doc_content, any other MCP tool

  TwitterPostReviewer / InstaPostReviewer:
    ✅ ALLOWED:   get_doc_content
    ❌ BLOCKED:   create_doc, any other MCP tool

  SkillBuilder (marketplace, zero-capability):
    ✅ ALLOWED:   nothing — it never touches an MCP tool. It only writes text
                  (a new specialist's instruction). Dispatch skips the grant
                  entirely for this skill — "nothing to lend" is the honest
                  encoding of "generates behavior, never invents tools."
    ❌ BLOCKED:   every MCP tool, always. A task that genuinely needs a NEW
                  tool the builder doesn't have access to → build fails
                  gracefully, nothing persisted. Marvis never invents tools.

  This read-vs-write, closed-tool-set split is the least-privilege scoping
  demonstration for BOTH the original marketplace and everything Marvis builds.

================================================================================
  GOOGLE DOCS SETUP (one-time, required for real doc creation)
================================================================================

By default, if OAuth hasn't been completed, create_doc will hang waiting for
browser consent (a URL is printed to the marvis server log).

1. Get a Google OAuth client:
   - Go to console.cloud.google.com → APIs & Services → Credentials
   - Create OAuth 2.0 Client ID (Desktop app)
   - Put the client id/secret in marvis/.env as GOOGLE_OAUTH_CLIENT_ID /
     GOOGLE_OAUTH_CLIENT_SECRET (this file is gitignored)

2. Set USER_GOOGLE_EMAIL in marvis/.env to the Google account you'll authorize.

3. First real create_doc call opens a consent URL (printed to the marvis
   server log) — open it, sign in, grant Docs + Drive access. The token is
   cached by the workspace-mcp server for subsequent calls.

4. If Ollama + the gdocs MCP round-trip is slow on your machine, raise the
   dispatch timeout in marvis/.env:
     DISPATCH_TIMEOUT_SECONDS=240
   (default is 120s; a cold Gemma generation + a first MCP call can exceed it)

  User wallet:      $500.00

  [BUILD — PIN #B "build_auth"]
  User wallet:      $497.50  (-$2.50)
  escrow:build-xxx: $2.50    (+$2.50)   ← locked during the build

  [SKILLBUILDER RUNS — writes instruction text only, no tool call]

  [VALIDATION GATE PASSES → build completion released]
  escrow:build-xxx: $0.00    (-$2.50)
  agent:SkillBuilder: $2.50  (+$2.50)   ← builder paid, ONE TIME

  [SKILL PERSISTED to app/owned-skills/linkedin-writer/ — permanent]

  [ORIGINAL TASK RE-RUNS on the now-owned skill — NO payment nodes at all]
  User wallet:      $497.50  (unchanged — the task itself is FREE)

  [EVERY SUBSEQUENT LinkedIn task, forever]
  User wallet:      unchanged — $0.00 every time, no PIN, no escrow

  [BUILD VALIDATION FAILS instead — fabricated tool / duplicate id]
  escrow:build-xxx: $1.00    (base kept — non-refundable)
  agent:SkillBuilder: $1.00  (base fee only)
  User wallet:      $498.50  (+$1.50 completion refunded)
  Nothing written to app/owned-skills/ — try again from scratch.

  Global ledger sum across ALL accounts: 0  (double-entry invariant, always)
  Every entry is hash-chained (SHA-256) — tamper-detectable.
  Verify any time: curl "localhost:8000/wallet/verify-chain?token=<TOKEN>"
